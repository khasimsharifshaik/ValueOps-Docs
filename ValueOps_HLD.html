1. Project Overview

What is ValueOps?

ValueOps is an Architecture as Code governance platform that implements the closed loop:

Discover → Design → Deliver → Run → Learn

The platform keeps strategy, architecture, delivery, runtime and feedback continuously aligned, modernising the architecture practice without replacing what engineering already does well.

One-Line Summary

SRC is observed → STD holds the rules → UX shapes the experience → APP enforces the rules → TGT is the workload that must satisfy them.

Program Context

  • Programme: Helio Modernisation
  • Delivery Partner: Virtusa CTO Advisory
  • Scope: MVP + R1 (Phases 1–5 & 12)
  • Personas: 8 (Product Manager, Enterprise Architect, Lead Architect, Solution Architect, Lead Engineer, Risk Officer, Ethics Reviewer, Compliance Lead)
  • Repository: github.com/chillisurfer-one/ValueOps-VirtusaX

2. Getting Started Guide

Prerequisites

  • Docker & Docker Compose
  • Node.js v22+
  • Git
  • PostgreSQL client tools (optional but recommended)

Quick Start Steps

  1. Clone the repository 
    git clone https://github.com/chillisurfer-one/ValueOps-VirtusaX.git
cd ValueOps-VirtusaX
  2. Start the Docker Compose stack 
    docker compose up -d

    This starts all services: PostgreSQL, Keycloak, Ollama, Weaviate, Redpanda, OPA, MinIO

  3. Set up environment variables 
    cp .env.example .env
# Edit .env with your values
  4. Initialize databases 
    npm run db:migrate
npm run db:seed
  5. Install dependencies & start the platform 
    npm install
npm run dev
  6. Access the platform
    • App: http://localhost:3000
    • Keycloak: http://localhost:8080 (realm: helio)
    • OPA: http://localhost:8181

Verify Installation

npm run test:governance  # Run governance gates
npm run validate:spec  # Validate spec linter
docker compose ps  # Check all services are running

Key Architectural Principles

🔒 Domain Separation

Five L1 domains with distinct authority; no cross-domain imports; communication via events & APIs.

📋 Policy & Control Separation

Policies govern design (OPA/Rego at authoring time). Controls govern build (tfsec/ArchUnit at PR/CI).

🔗 Dual-Write Audit

Every governance action writes: (1) artefact state change + (2) immutable audit record. Both writes required; partial write = failure.

🎯 Spec-Driven Development (SDD v0)

Intent specs (problem, actors, outcome, behaviour, constraints, acceptance) → 1:1 per story → signed-off by LE → governs build.

5. Core Components & Capabilities

D1 — Architecture as Code & Continuous Alignment (Spine)

📝 D1A — Authoring & Artefacts

Four artefact types: ADRs (context, decision, alternatives, consequences) · Models (C4 Container, Deployment as PlantUML) · Intent Specs (SDD v0) · Jira Stories with acceptance criteria.

🛡️ D1B — Policies & Controls as Code

Policies (OPA/Rego, YAML) — design-time, govern intent. Controls (tfsec, ArchUnit, Conftest) — build-time, at PR/CI.

⚙️ D1C — Governance & Workflow

Chain: SA drafts → LA reviews & recommends → EA approves → binding in CI/CD. LE accepts at handover or raises variation. Waiver loop with full traceability.

🔍 D1D — Drift & Conformance

Drift detection: implementation vs reference architecture vs intent. ADR → implementation traceability. Conformance checks in CI; failing gate blocks merge.

🏷️ D1E — Metadata & Taxonomy

Machine-readable ADR metadata linking decisions to strategy and enforcement. Lifecycle tags: from-strategy, scaffolded, awaiting-ADR-review. Tags drive pipeline policy resolution.

🔧 D1F — Tooling & Pipelines

GitHub + Jira integration. GitHub Actions guardrails (policy compliance, drift, traceability). Pre-commit + PR enforcement. Scaffold generation from accepted ADR + spec.

D3 — Risk, Ethics & Trust Architecture (Assurance Backbone)

📊 D3A–D3B — Risk as Code

Nested risk taxonomy, propagation modelling, multi-touchpoint vectors, contextual scoring. CI guardrail automation (flag/block PRs, waivers). ISO 31000, NIST RMF, FCA alignment.

✅ D3C — Ethics as Code

Ethics taxonomy (fairness, autonomy, transparency, inclusion, accountability, well-being). Context-aware policy binding by jurisdiction/use case. Ethics-aware tests, explainability hooks. FCA, ICO, EU AI Act exports.

🔐 D3H — Privacy-as-Code

Data minimisation with documented legal basis. Privacy guards / schema validation. Consent flows. DPIA support. Privacy constraints expressed as outcome-shaped ADR decisions.

6. Data Architecture

Schema Design

Schema Purpose Domains Status (MVP+R1)
d1 Architecture as Code (authoring, policies, controls, governance) D1A–D1F + D1G metadata ✅ Populated
d2 Strategy & Capability Intelligence (stub: problem capture → epic/story generation) D2 stub only ✅ Populated (minimal)
d3 Risk, Ethics & Trust (risk taxonomy, ethics policies, privacy constraints) D3A–D3B, D3C, D3H ✅ Populated
d4 Context / Telemetry Sensing (deferred to R2) D4A–D4H ❌ Reserved
d5 Learning / Innovation (deferred to R2) D5A–D5F ❌ Reserved
audit Immutable, append-only audit trail (hash-chained, tamper-evident) All domains ✅ Critical (MVP)

Key Data Principles

🔒 No Cross-Schema Foreign Keys

Domain separation enforced in the data layer. Cross-domain references are by ID over events/APIs, not FKs.

🔄 Reversible Migrations

Managed by Prisma; versioned and reviewed. No destructive changes (drop/rename) without a governed gate.

🔗 Append-Only Audit Trail

Hash-chained, tamper-evident: each record carries prevHash. Writer roles hold INSERT only (no UPDATE/DELETE).

🎯 Dual-Write Pattern

Every governance action writes: (1) artefact state change (versioned commit to STD or spec status update) + (2) audit event. Both required; partial write rolls back.

Event Architecture (Redpanda)

Topic Convention: valueops.<domain>.<event> Examples: valueops.d1.adr-drafted valueops.d1.adr-recommended valueops.d1.adr-approved valueops.d1.spec-signed-off valueops.d1.handover-accepted valueops.d3.risk-waived valueops.d3.ethics-flagged valueops.d3.compliance-attested Event Schema: id · timestamp (ISO 8601) · actor · action · artefactRef · decision · rationale (required for waivers)

Integration Pattern

  • Within a domain: Postgres (schema-scoped) + events for broadcast
  • Cross-domain: Events via Redpanda; consumers apply domain logic and update their schema
  • External (TGT): Conformance engine reads STD policies; applies them against TGT artefacts via CI

⚠️ Self-Hosted AI Only: Ollama on-box/adjacent. No external model calls at runtime. No PII/customer data to the model.
⚠️ Apple Silicon: Pin OPA to platform: linux/amd64 in Compose.

10. Key Principles & Non-Negotiables

Foundational Constraints

🚫 APP ≠ TGT

ValueOps-VirtusaX (APP) is the governance platform. ValueOpsVX-Banking-Target (TGT) is a governed workload. Platform code never goes in TGT; workload code never goes in APP.

🔐 Domain Model is Frozen

Five L1 domains with fixed authority and frozen L1–L3 catalogue in STD. Do not extend, rename, or blur boundaries. A new domain = escalation, not commit.

📚 Catalogue is Single Source of Rules

Rules live in STD (via the chain: SA → LA → EA). APP enforces; TGT complies; SRC is evidence; UX is experience.

🎯 Spec is Intent; Story is Anchor

One intent spec per story, 1:1. Intent only (problem, actors, outcome, behaviour, constraints, acceptance). No mechanism, no bindings. Governance lives on the ADR.

🛡️ Policies Govern Design; Controls Govern Build

A guardrail at design-time = policy (OPA/Rego, YAML). A guardrail at PR/CI = control (tfsec/ArchUnit/Conftest). Never conflate.

🔗 Every Action is Recorded (Dual-Write)

Approvals (approve/recommend/sign-off/accept-handover/waive) update the artefact state AND append the audit trail. Both writes required; partial = failure.

🔒 AI Accelerates; Humans Decide

D1G (Ollama) assists authoring; nothing it produces binds without human approval and pipeline validation. No external model calls; no PII to the model.

👥 Persona Surfacing is Server-Side Enforced

Business personas (PM, Risk Officer, Ethics Reviewer, Compliance Lead) see plain business language only — no Rego, no policy code, no ADR metadata. Enforced server-side by a projection layer; a test fails if violated.

🏗️ One Stack Per Repo

APP = NestJS/TypeScript. Do not introduce a second backend stack.

📊 No Real Data Anywhere

Synthetic data only (dev, demo, shared EC2, logs, prompts). No customer or personal data, ever.

Out of Scope Until R2+

Do NOT build: Full D2, D4, D5, D1H, graph store (reuse graph), D2G (capability-investment engine), admission-time cluster enforcement, service mesh.

Mandatory Testing Gates

  • ✅ Persona surfacing — business personas cannot retrieve policy code or ADR metadata
  • ✅ Approval authority — personas without rights cannot approve
  • ✅ Audit dual-write — approval writes both artefact and trail; partial write fails
  • ✅ Spec linter — mechanism language rejected; 1:1 storyId; no orphans
  • ✅ ADR coverage — `gap` blocks handover; `references` or `none_required` only
  • ✅ Audit immutability — no UPDATE/DELETE path on audit schema; chain verification passes
