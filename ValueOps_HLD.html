1. Project Overview

ValueOps is the authoring and governance platform within the Architecture as Code paradigm. It makes live the closed loop:

    Discover  ──>  Design  ──>  Deliver  ──>  Run  ──>  Learn
        ^                                                  │
        └──────────────────────────────────────────────────┘
Figure 1 — The ValueOps Continuous Alignment Loop

The platform keeps strategy, architecture, delivery, runtime and feedback continuously aligned rather than drifting between documents and reality. It modernises the architecture practice; it does not replace what engineering already does well.

In one line: SRC is observed → STD holds the rules → UX shapes the experience → APP enforces the rules → TGT is the workload that must satisfy them.

2. Getting Started

Prerequisites

  • Node.js v22 (LTS) & npm
  • Docker & Docker Compose (substrate services)
  • Git access to ValueOps-VirtusaX and related repos
  • Apple Silicon note: pin OPA to platform: linux/amd64

Quick Start

# 1. Clone and install
git clone https://github.com/chillisurfer-one/ValueOps-VirtusaX.git
cd ValueOps-VirtusaX
npm install

# 2. Stand up the substrate (day one)
docker compose up -d

# 3. With observability (optional, off by default)
docker compose --profile observability up -d

# 4. Run database migrations
npx prisma migrate deploy          # or: npm run db:migrate

# 5. Seed synthetic banking-domain data
npm run db:seed                     # SYNTHETIC data only — no real customer data

# 6. Verify Keycloak
# Browse to http://localhost:8080 — confirm "helio" realm with persona roles

# 7. Start the platform
npm run start:dev

4. Domain Model (Frozen L1–L3)

Five L1 domains with distinct, non-overlapping authority. The frozen L1–L3 model is the source of truth in STD. Do not blur these lines, rename entries, or add capabilities beyond the catalogue.

Separation of concerns (invariant): Architecture control → D1 · Decisions → D2 · Constraints → D3 · Context/telemetry → D4/D4H · Learning/innovation → D5 · AI synthesis → D1G · Runtime control plane → D1H. Of these, only D1 (D1A–D1G), D2 stub, and D3 land in MVP→R1.

5. The Four Artefacts & the Mechanism

The ADR declares intent and carries the policy/control bindings; the linked policy checks the design and the control checks the build; divergence is flagged as drift and traced back to the governing ADR.

ADR — Decision of Record

Context, decision, alternatives, consequences. Immutable once approved. Carries metadata linking up to strategy and out to enforcement. Governs why the target is shaped as it is.

Model / Visual Artefact

Executable design — C4 / structured diagrams as versioned artefacts, not static images. Governs design expression. Tooling: PlantUML.

Policy — Governs Design

What must be true at design time. Policies govern design (intent). Expressed as OPA/Rego or YAML. Stored in /policies/.

Control — Governs Build

How compliance is proven at build time (PR/CI). Controls govern build (enforcement). Tools: tfsec, ArchUnit, Conftest. Stored in /controls/.

Figure 5 — PostgreSQL Schema Separation (one schema per domain + audit)

Data Separation Rules

  • No cross-schema foreign keys. Domain separation is enforced in the data layer; cross-domain references are by id over events/APIs, not FKs.
  • Prisma migrations: reversible, versioned, reviewed. No destructive change (drop/rename) without a governed gate.
  • Migrations run in CI, not by hand on shared hosts.
  • Synthetic data only — no real customer or personal data anywhere.

Audit Schema Detail

Append-only, Hash-chained (ADR-0002, proposed)

  • Each record carries prevHash = hash of the prior record — tamper-evident.
  • Writer roles (CI token, APP audit role) hold INSERT only — no UPDATE / DELETE grant.
  • Any edit or deletion breaks the chain and is detectable.
  • Deferred hardening: periodically anchor the latest chain hash out-of-band.

Audit Event Schema

interface AuditEvent {
  eventId:      string;          // UUID
  timestamp:    string;          // ISO 8601
  actor:        string;          // user id + persona
  action:       AuditAction;     // see below
  artefactRef:  string;
  artefactType: string;
  storyId?:     string;
  adrRefs:      string[];
  decision:     'pass' | 'fail' | 'waiver' | 'approve' | 'reject';
  rationale?:   string;          // required for waiver/override
  policyRefs:   string[];
  commitSha:    string;
  runId?:       string;
  prevHash:     string;          // hash of prior record (chain integrity)
}

type AuditAction =
  | 'policy-eval' | 'control-check'
  | 'adr-transition' | 'spec-signoff'
  | 'handover' | 'waiver' | 'override';

Redpanda Event Topics

ConventionExamples
valueops.<domain>.<event>valueops.d1.adr-approved · valueops.d3.risk-waived · valueops.d1.spec-signed-off

Event schemas are versioned. Events carry ids + timestamps — do not assume last-write-wins.

10. Audit Trail

The immutable audit trail is the assurance backbone. Two emitters, one schema, one sink.

Emitter A — CI (GitHub Actions)

Writes an event at each pipeline evaluation: policy-eval, control-check, pass/fail/waiver. Every evaluation is recorded regardless of outcome.

Emitter B — APP / UX Approvals

At each human governance action (approve, recommend, sign-off, waive, override), APP performs a dual write: (1) update artefact governance state, (2) append audit event. Both writes required — partial write rolls back.

Dual-Write Mechanism

  Persona triggers approval action
           │
           ▼
  ┌────────────────────────┐
  │  Authority Check       │  Does persona hold this right? (EA approves, LA recommends, etc.)
  └───────────┬────────────┘
              │ yes                      │ no
              ▼                          ▼

  │  Write 1: Update   │      │  Reject & audit the  │

  │  (version-control) │      └─────────────────────┘
  └─────────┬──────────┘
            │

  Either fails?  → Roll back / alarm (never silent success)
Figure 7 — Dual-Write: Artefact State + Audit Event

Trust Boundary

  • CI and UX are the only writers. No other path writes to the audit trail.
  • APP owns the read/export, not the write. D3 assurance surfaces query and export.
  • Export formats aligned to: ISO 31000, NIST RMF, FCA, ICO, EU AI Act.

11. AI-Layer Governance (D1G)

AI-Assisted and Governed

D1G accelerates authoring; nothing it produces is binding until the human chain approves and the pipeline validates. AI-drafted ADR rationale is a suggestion until the author owns it; the decision-trace annotator versions the human edit.

Self-Hosted Only

Ollama on-box/adjacent; no external model endpoint at runtime. The platform never sends data off the substrate to a model. Model pinned (name + version) for reproducibility; changing it is an ADR.

D1G Capabilities (R1, Phase 12)

  • Contextual insight surfacing — related ADRs/policies/reusable components by tag; inline risk/ethics/platform rules
  • Intelligent pattern matching & reuse with fitness scoring
  • Decision intelligence & guardrail enforcement — AI-assisted ADR drafting with traceable rationale, trade-off views, pre-merge OPA/Rego validation
  • Collaboration & knowledge amplification — NL explanation of ADRs/diagrams/policies, AI-suggested rationale
  • Decision-trace annotator — versioned, auditable
  • Telemetry feedback loop — accept/edit/ignore (PII-safe)
  • Persona-adaptive, progressive-disclosure UI
No PII / customer data to the model. Prompts are built from governed artefacts and metadata, not raw customer data (D3H).

14. Mandatory Test Gates

A red governance-gate test blocks merge. These are non-negotiable.

#GateWhat It VerifiesBlocks
1 Persona Surfacing A business-persona token (PM, Risk Officer, Ethics Reviewer, Compliance Lead) cannot retrieve a policy body or ADR metadata field via any endpoint. Merge
2 Approval Authority A persona without the right cannot approve. EA approves; LA recommends; SA drafts; LE handover; PM approves nothing architectural. Merge
3 Audit Dual-Write An approval writes both artefact state and audit trail. A partial write (artefact-only or trail-only) fails. Merge
4 Spec Linter Mechanism language rejected; 1:1 storyId; no orphans; additionalProperties: false; ADR coverage consistent. Merge
5 Gap Blocks Handover A spec with adrCoverage: 'gap' blocks handover until resolved to references or none_required. Handover
6 Audit Immutability No UPDATE/DELETE path on the audit schema. Hash-chain verification passes. Merge

15. Key Principles

APP is not TGT

The platform (APP) governs the workload (TGT). Platform domain code (D1–D5) never goes in TGT; TGT never carries platform code.

Frozen Domain Model

Five L1 domains with fixed authority, frozen L1–L3 catalogue in STD. Do not extend, rename, or blur boundaries.

Catalogue is Single Source

Rules live in STD; APP enforces; TGT complies; SRC is evidence; UX is the experience. Author through the chain (SA → LA → EA), never free-type.

Spec is Intent (SDD v0)

One spec per story, 1:1, mandatory. Intent only — no mechanism, no interfaces, no technology. Governance lives on the ADR, never in the spec.

Policies Govern Design; Controls Govern Build

A guardrail at design time is a policy (OPA/Rego, YAML). One at build time is a control (tfsec/ArchUnit/Conftest at PR/CI). Never conflate.

Dual-Write Audit

Every governed action updates the artefact state and appends to the immutable audit trail. Both writes required — a partial write is a failure.

AI Accelerates, Humans Decide

Nothing the AI layer produces is binding until the human chain approves and the pipeline validates. Self-hosted only (Ollama); no external model calls.

Persona Surfacing is Server-Side

Business-persona DTOs exclude Rego, ADR metadata, and policy code by construction. Enforced at the API boundary, never the client. Tested with a mandatory gate.

One Stack per Repo

APP = NestJS/TypeScript (ADR-0001). Do not introduce a second backend stack. Do not import across domain boundaries in code.

Synthetic Data Only

No real customer or personal data anywhere — dev, demo, shared EC2, prompts, logs. Enforced with gitleaks in pre-commit and CI.

16. Guardrails — What Not to Do

These are hard boundaries. A seeming need to cross one is an escalation, not a commit.

Repo boundariesDo not put platform (D1–D5) code in TGT, workload code in APP, or rules/workload code in STD.

ScopeDo not build R2/R3 scope — no full D2, D4, D5, D1H, graph store, D2G, admission control, or service mesh.
