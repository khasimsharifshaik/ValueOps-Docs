🔄 Integration Layer

  • Event Broker (Redpanda) Topics: valueops.d1.*, valueops.d3.*, audit topic. Event schema versioning.
  • Event Emitters D1-D3 domain services publish events on state changes (ADR approved, risk waived, etc.)
  • Audit Trail Emitter Dual-write handler: (1) artefact state change + (2) audit event append
  • OPA Service (Runtime) Long-running OPA instance for in-app policy evaluation, decision logging
  • Conformance Engine Reads STD catalogue, applies policies/controls to TGT via CI (GitHub Actions)
  • GitHub Actions Orchestrator Rule-validation pipeline trigger, OPA CLI executor, audit event emitter

💾 Data & Infrastructure Layer

  • Prisma ORM Schema definitions, reversible migrations, type safety, seed scripts
  • Keycloak (Auth) Realm: helio. Roles: PM, EA, LA, SA, LE, Risk Officer, Ethics Reviewer, Compliance Lead
  • MinIO (Object Store) Artefact versioning, backup, PlantUML source storage
  • Weaviate (Vector Store) Semantic embeddings for D1G (contextual insight surfacing, pattern matching)

  • OPA CLI (Policy Engine) Runs in CI/CD runners. Executes Rego policies, emits evaluation results.
  • tfsec, ArchUnit, Conftest Build-time controls (IaC, code, config validation)

📊 Key Data Flows

1. ADR Authoring & Approval Flow (D1C): SA drafts ADR (Frontend) → Backend D1:Governance validates → Event: adr-drafted (Redpanda) ↓ LA reviews & recommends (UX action → dual-write) → Artefact state: recommend_by_LA (Postgres d1) → Audit event: adr-recommended (Postgres audit + Redpanda) ↓ EA approves (UX action → dual-write) → Artefact state: approved_by_EA (Postgres d1) → Audit event: adr-approved (Postgres audit + Redpanda) → GitHub Actions triggered → OPA evaluates against STD policies → TGT repo conformance checked ↓ LE accepts at handover → Builds conform to approved ADR (or raises variation) 2. Intent Spec Workflow (SDD v0, D1A): SA creates spec (SpecAuthoringPanel) → Real-time linter (frontend + backend) → Spec status: drafted (Postgres d1) ↓ Spec linter validates (no mechanism, outcome-shaped acceptance) → Status: valid (gates handover if all required sections present) ↓ LE sign-off (SpecViewer UX) → Dual-write: status: signed_off + baselinedAt + baselineVersion → Audit event: spec-signed-off (Postgres audit + Redpanda) → Governance gate: ADR coverage resolved (no gap) ↓ Build phase: LE accepts handover or raises variation → Variation → re-sign (Postgres d1) + LA approval 3. Conformance & Drift Detection (D1D): TGT repo commits code → GitHub Actions triggered (push/PR) → Rule-validation pipeline fetches applicable policies from STD → OPA evaluates design artefacts (ADRs, models, policies) via CLI → tfsec/ArchUnit/Conftest run build checks → Conformance engine compares TGT vs reference architecture (from ADR) → Drift detected → flagged + traced to governing ADR → Audit event: control-check pass/fail/waiver (Postgres audit + Redpanda) → Merge blocked if critical controls fail (or human waiver with rationale) 4. Risk & Ethics Governance (D3): Risk Officer flags risk (D3:Risk service) → Risk event: risk-flagged (Redpanda) → Risk Waiver action (UX) → dual-write → Artefact: risk waiver recorded + rationale mandatory → Audit event: risk-waived (Postgres audit + Redpanda) ↓ Ethics Reviewer reviews ethics flag → Ethics event: ethics-flagged (Redpanda) → Compliance Lead attests (UX action) → Audit event: ethics-attested (Postgres audit + Redpanda) 5. Persona-Scoped Surfacing (Server-Side Projection): User logs in via Keycloak (helio realm) → JWT token carries persona role → Backend auth service verifies (JWT + Keycloak introspect) → Request hits personas/projection layer → Projection filters DTO fields (policy code, ADR metadata excluded for PM/Risk/Ethics/Compliance) → Frontend receives filtered response only → Governance test: PM token cannot retrieve ADR metadata → test fails if breached

🔌 Communication Patterns

Synchronous (HTTP/REST):
  • Frontend ↔ Backend API (JWT auth)
  • Backend ↔ Keycloak (token validation)
  • Backend ↔ OPA (policy decisions)
  • Backend ↔ Weaviate (semantic search)
Asynchronous (Events):
  • D1–D3 services publish domain events
  • Audit emitter publishes to audit topic
  • Cross-domain event consumers
  • CI/CD pipeline event subscriptions
Database (Direct):
  • NestJS services via Prisma ORM
  • Schema isolation (d1–d3 + audit)
  • No cross-schema FKs (event-based)
  • Audit trail append-only (INSERT only)
