■ Layer 3 — Integration Components ▼
Event Bus (Redpanda) Async messaging backbone. Topics follow valueops.<domain>.<event> convention (e.g. valueops.d1.adr-approved, valueops.d3.risk-waived). Versioned event schemas.
Consumers: all backend domain modules. Producers: D1C, D3A-B, Audit Emitter
Event Emitters Per-domain event publishers. Events carry ids + timestamps; no last-write-wins assumption. Cross-domain communication contract.
Depends on: Redpanda client library, versioned event schemas
Audit Emitter (CI-side) GitHub Actions writes audit events at each pipeline evaluation: policy-eval, control-check, pass/fail/waiver. Same schema as app-side emitter.
Depends on: GitHub Actions runner, PostgreSQL (audit schema, INSERT-only token)
OPA Service Long-running OPA service in the substrate for in-app/runtime policy evaluation. Same Rego policies as CI — do not diverge rule sources.
Depends on: STD policies (mounted/synced), OPA runtime
Conformance Engine Reads the STD catalogue and applies policies (design-time) and controls (build-time) against TGT. Divergence flagged as drift, traced to the governing ADR.
Depends on: STD repo, TGT repo, OPA Service, GitHub API
GitHub Actions Orchestrator Rule-validation pipeline: trigger → policy resolution (tag → selector) → design-time policy eval → build-time control check → audit emit. Three sites: TGT, APP, STD.
Depends on: OPA CLI, tfsec, ArchUnit, Conftest, gitleaks
■ Layer 4 — Data & Infrastructure Components ▼
PostgreSQL Schemas: d1, d2, d3 (business domains) + audit (append-only, hash-chained — ADR-0002). No cross-schema FKs — domain separation enforced in the data layer.
Audit schema: INSERT only, no UPDATE/DELETE grants. Writer roles: CI token, APP audit role.
Prisma ORM Reversible, versioned migrations reviewed in PR. No destructive change (drop/rename) without a governed gate. Migrations run in CI, not by hand.
Depends on: PostgreSQL, NestJS Prisma module
Keycloak Realm: helio. 8 roles mapped to personas (PM, EA, LA, SA, LE, Risk Officer, Ethics Reviewer, Compliance Lead). RBAC + the projection layer enforces surfacing.
Provides: OIDC tokens, JWKS endpoint, role claims
MinIO S3-compatible object store for artefacts: ADR documents, PlantUML models, intent specs, scaffold templates, compliance exports.
Consumers: D1A Authoring, D1F Tooling, D3 compliance export
Weaviate Vector store for D1G Cognitive Workbench. Semantic search over governed artefacts, related ADR/policy surfacing, pattern matching & reuse scoring.
Fed by: D1A (artefact embeddings), D1E (metadata). Queried by: D1G assistant.
OPA CLI (CI runner) opa eval / conftest inside the GitHub Actions runner. Self-contained policy evaluation for CI pipeline. Same Rego as the OPA Service.
Depends on: STD policies (checked out), GitHub Actions runner environment
Ollama Self-hosted AI model server, pinned model (name + version). No external model calls at runtime. Prompts built from governed artefacts, not customer data. PII-safe.
Consumers: D1G Cognitive Workbench. Changing the model requires an ADR.
GitHub Source control, pull request workflow (the governance chain runs as PR checks), GitHub Actions CI, webhooks for event triggers.
Integrates with: all layers — source of truth for version-controlled artefacts

◆ Key Data Flows

1 ADR Authoring & Approval (SA draft → LA recommend → EA approve → dual-write audit) ▼
1. SA Creates ADR Draft
Solution Architect authors the ADR in the SA Surface (context, decision, alternatives, consequences). D1G may assist with AI-suggested rationale (R1). Status: draft.
SA
2. ADR Persisted to STD
Backend D1A module saves the ADR to PostgreSQL (d1 schema) and stores the document in MinIO. Metadata tags (D1E) are applied for policy resolution.
SYSTEM
3. LA Reviews & Recommends
Lead Architect reviews the ADR on the LA Surface. Authority check confirms LA role before the transition. On recommendation, status moves to recommended. Dual-write: artefact state updated + audit event appended (prevHash chained).
LA
4. EA Approves
Enterprise Architect approves the ADR on the EA Surface. Authority check confirms EA role. Status moves to approved — decision content is now immutable. Dual-write: version-controlled change to STD + hash-chained audit event.
EA
5. Event Emitted & Pipeline Triggered
Event valueops.d1.adr-approved published to Redpanda. GitHub Actions picks up the change: policies validated, controls checked. A second audit event records the pipeline result (pass/fail). ADR is now binding in CI/CD.
CI
6. Binding — Enforcement Active
The approved ADR's linked policies and controls are now enforced at design-time and build-time respectively. Any future TGT commit that violates the ADR's policies is blocked at PR.
SYSTEM
2 Intent Spec Workflow (SDD v0: create → lint → LE sign-off → handover gate) ▼
1. SA Authors Intent Spec
Solution Architect creates the spec in SpecAuthoringPanel. One spec per story (1:1). Fields: problem, actors, outcome, behaviour, constraints, acceptance criteria. Status: drafted.
SA
2. Spec Linter Validates
Automated linter checks: additionalProperties: false, all required sections present, storyId unique, acceptance outcome-shaped (rejects mechanism language: endpoint, /api, REST, GET/POST, schema/table names), ADR coverage consistent. A failed linter blocks valid.
LINTER
3. Status → Valid
On passing the linter, the spec status transitions to valid. ADR coverage must be references or none_required — a gap blocks the handover gate.
SYSTEM
4. LE Reviews in SpecViewer
Lead Engineer reviews the validated spec in SpecViewer. The spec is a contract — LE accepts or raises a variation before build. This is the handover seam.
LE
5. LE Sign-off (Audited Approval)
Authority check confirms LE role. Dual-write: sets status: signed_off + baselinedAt + baselineVersion on the spec AND appends a hash-chained audit record. Status: signed_off.
LE
6. Handover Gate Passes
Gate checks: 100% intent coverage (every story has a valid spec), no orphans, no gap ADR coverage, LE signed off. Build proceeds. Post-baseline changes re-enter via the variation loop.
GATE
3 Conformance & Drift Detection (TGT commit → GitHub Actions → OPA eval → audit) ▼
1. TGT Commit / PR Created
A developer pushes a change to the target workload repository (ValueOpsVX-Banking-Target). The PR event triggers the rule-validation pipeline.
DEVELOPER
2. Policy Resolution
GitHub Actions resolves which policies apply from the changed artefact's metadata/taxonomy tags (D1E). Tag → policy-selector mapping. Unresolvable/untagged artefacts fail closed.
CI
3. Design-time — Policy Evaluation
OPA CLI (opa eval / conftest) evaluates design artefacts against applicable policies from STD. Reported separately from controls. Fail = hard block on merge.
OPA CLI
4. Build-time — Control Checks
tfsec (IaC), ArchUnit (Java architecture in TGT), Conftest run over the build. Controls check that the build conforms to the governing ADR. Reported separately from policies.
CI CONTROLS
5. Drift Analysis
Conformance engine compares implementation against reference architecture and intent. Divergence flagged as drift, traced to the governing ADR. A failing gate is real, not advisory.
CONFORMANCE
6. Audit Event Emitted
Every evaluation (pass/fail/waiver) writes an immutable audit event: artefact ref, policyRefs[], decision, actor, commit SHA, run ID, linked ADR, prevHash.
CI AUDIT
4 Risk & Ethics Governance (risk flag → waiver with mandatory rationale → compliance attestation) ▼
1. Risk / Ethics Flag Raised
An artefact (ADR, spec, story, code) is annotated with @risk or @ethics. The flag may be raised automatically by the CI pipeline or manually by an architect. Event published to Redpanda.
SACI
2. Risk Officer Reviews
Risk Officer sees the flag on the Risk Surface (plain business language — no Rego, no policy code). Contextual risk scoring: criticality, exposure, likelihood. Heatmap view. Decision: accept, escalate, or waive.
RISK OFFICER
3. Ethics Reviewer Assesses
Ethics Reviewer reviews @ethics flags on the Ethics Surface (plain business language). Taxonomy: fairness, autonomy, transparency, inclusion, accountability, well-being. Context-aware policy binding by jurisdiction/use case.
ETHICS REVIEWER
4. Waiver / Override (Mandatory Rationale)
If a waiver is requested: authority check confirms the persona holds waiver rights (Risk Officer for risk, Ethics Reviewer for ethics). Mandatory rationale is required — no silent waivers. Dual-write: artefact state updated + audit event with rationale.
RISK OFFICERETHICS REVIEWER
5. Compliance Attestation
Compliance Lead reviews the complete risk/ethics trail on the Compliance Surface (plain business language). Attests that the evidence is complete. Exports aligned to ISO 31000, NIST RMF, FCA, ICO, EU AI Act.
COMPLIANCE LEAD
6. Immutable Audit Trail
Every risk/ethics action is recorded in the hash-chained audit trail: flags, assessments, waivers (with rationale), attestations, rejected/unauthorised attempts. Tamper-evident — any edit breaks the chain.
AUDIT
5 Persona Surfacing (login → JWT → projection layer → filtered DTO) ▼
1. User Authenticates via Keycloak
User logs in through the Keycloak helio realm. Keycloak issues a JWT with persona role claims (one of: PM, EA, LA, SA, LE, Risk Officer, Ethics Reviewer, Compliance Lead).
KEYCLOAK
2. API Client Sends Request
The React frontend API Client attaches the JWT as a Bearer token. On 401, the client refreshes the token automatically. Request hits the NestJS gateway.
FRONTEND
3. Auth Service Validates Token
NestJS Auth Service validates the JWT against Keycloak's JWKS endpoint. Extracts the persona role. Rejects expired or tampered tokens.
AUTH SERVICE
4. Personas Projection Layer Filters
Server-side DTO filtering at the API boundary. Business personas (PM, Risk Officer, Ethics Reviewer, Compliance Lead) receive responses with Rego bodies, ADR metadata, and policy code excluded by construction. The fields are not serialised — not merely hidden in the UI.
PROJECTION
5. Persona Surface Renders
The frontend renders the persona-appropriate surface. Technical personas (EA, LA, SA, LE) see the full architecture. Business personas see plain language only. Available actions match persona rights (e.g. PM has no ADR buttons).
PMEARISK
6. Governance Gate Test
Mandatory test: a business-persona token cannot retrieve a policy body or an ADR metadata field via any endpoint. A red test blocks merge. This is mechanism, not aspiration.
TEST

◆ Communication Patterns

SYNC Synchronous (HTTP / REST)

  • Frontend ↔ Backend — REST API with JWT Bearer tokens
  • Backend → Keycloak — OIDC token validation (JWKS endpoint)
  • Backend → OPA Service — Runtime policy evaluation (/v1/data)
  • Backend → Ollama — D1G AI-assisted authoring (/api/generate)
  • Backend → Weaviate — Semantic search queries (GraphQL)
  • Backend → MinIO — Artefact read/write (S3 API)
  • D1F Tooling → GitHub API — Scaffold generation, PR creation
  • D1F Tooling → Jira API — Epics/Stories synchronisation

ASYNC Asynchronous (Events)

  • D1C → Redpanda — valueops.d1.adr-approved
  • D1C → Redpanda — valueops.d1.spec-signed-off
  • D3A-B → Redpanda — valueops.d3.risk-waived
  • D3C → Redpanda — valueops.d3.ethics-flag-raised
  • Audit → Redpanda — valueops.audit.event-recorded
  • GitHub → Actions — Webhook-triggered CI pipelines
  • Cross-domain communication uses events only — no direct imports

DB Database (Prisma ORM)

  • D1 modules → d1 schema — ADRs, specs, models, metadata
  • D2 module → d2 schema — Spec capture (stub)
  • D3 modules → d3 schema — Risk, ethics, privacy
  • Audit Emitter → audit schema — INSERT only, hash-chained
  • No cross-schema FKs — domain separation enforced in the data layer
  • Cross-domain refs by ID over events/APIs, not foreign keys
  • Migrations — reversible, versioned, reviewed in PR, run in CI

◆ Docker Compose Deployment Topology

Note: Same Compose stack runs locally and on the shared EC2 host. Observability services are behind an optional profile (--profile observability), off by default. All stateful services on the EC2 host use a dedicated EBS volume, snapshot-backed. HTTPS via TLS-terminating reverse proxy on port 443.
⚙ valueops-app :3000

NestJS platform backend. Node v22. Module-per-domain (src/d1, src/d2, src/d3). REST API + WebSocket gateway. Entry point for all governed actions.

⚙ valueops-frontend :8080

React + TypeScript SPA. 8 persona surfaces. SpecAuthoringPanel, SpecViewer. Served via nginx or dev server.

⚙ postgres :5432

PostgreSQL database. Schemas: d1, d2, d3, audit. Audit schema: append-only, hash-chained, INSERT-only grants. EBS-backed on EC2.

⚙ redpanda :9092

Redpanda (Kafka-compatible). Event backbone. Topics: valueops.<domain>.<event>. Versioned event schemas. EBS-backed on EC2.

⚙ keycloak :8180

Keycloak identity provider. Realm: helio. 8 persona roles. OIDC/JWKS. Single front door for the shared environment.

⚙ opa :8181

Open Policy Agent service. Runtime policy evaluation for in-app use. Same Rego as CI. Apple Silicon: pin platform: linux/amd64.

⚙ ollama :11434

Self-hosted AI model server. Pinned model (name + version). D1G Cognitive Workbench. No external model calls. PII-safe prompts only.

⚙ minio :9000

MinIO S3-compatible object store. Artefact storage: ADRs, models, specs, scaffold templates, compliance exports. EBS-backed on EC2.

⚙ weaviate :8090

Weaviate vector store. Semantic search for D1G. Artefact embeddings, related ADR/policy surfacing, pattern matching. EBS-backed on EC2.

⚙ prometheus :9090

Metrics collection. Scrapes OTel endpoints from all services. Retention on EBS.

profile: observability
⚙ grafana :3001

Dashboards and visualisation. Pre-configured dashboards for platform health, governance metrics, audit trail statistics.

profile: observability
⚙ loki :3100

Log aggregation. Structured JSON logs with correlation/trace IDs. PII-safe — no customer/personal data in logs.

profile: observability
